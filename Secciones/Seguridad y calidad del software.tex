\chapter{Seguridad y calidad del software}
\label{Seguridad y calidad del software}

\section{Posibles vulnerabilidades}

Dada la naturaleza móvil, web y colaborativa de la aplicación, el sistema está expuesto a diversas amenazas operativas y cibernéticas. Se han identificado las principales vulnerabilidades específicas del negocio y se proponen sus respectivas mitigaciones técnicas

\begin{enumerate}
	\item \textbf{Robo de Cuentas y Suplantación de Identidad (Account Takeover)}
	\begin{itemize}
		\item Vulnerabilidad: Acceso no autorizado a cuentas de Clientes, Transportistas o Administradores mediante ataques de fuerza bruta, phishing o interceptación de credenciales. En el caso de los transportistas, la suplantación puede derivar en el robo de mercancía.
		\item Medida de Mitigación: Autenticación Robusta: Implementación de tokens de acceso seguros (JWT - JSON Web Tokens) para mantener las sesiones activas en los dispositivos móviles de forma encriptada.
		\begin{itemize}
			\item Políticas de Contraseñas: Restricciones en el backend que obliguen al uso de contraseñas complejas (mínimo 8 caracteres, mayúsculas, minúsculas, números y caracteres especiales).
			\item Verificación de Identidad: Integración con servicios de validación biométrica o validación manual rigurosa por parte del Administrador mediante la revisión de la cédula de identidad y antecedentes legales antes de activar al conductor.
		\end{itemize}
	\end{itemize}
	
	\item \textbf{Manipulación de Datos GPS y Spoofing de Ubicación }
	\begin{itemize}
		\item Vulnerabilidad: Modificación maliciosa de las coordenadas de geolocalización enviadas por el dispositivo del transportista. Esto podría usarse para simular rutas falsas, cobrar tarifas erróneas o encubrir desvíos durante el traslado.
		\item Medida de Mitigación:
		\begin{itemize}
			\item Validación de Origen de Coordenadas: Filtros en la API que analicen la coherencia matemática de la velocidad y trayectoria del transportista entre cada actualización de los 2.67 segundos.
			\item Cifrado de Telemetría: Todo el flujo de datos de posicionamiento global transmitido desde la app móvil hacia el servidor web se canalizará de forma obligatoria mediante el protocolo seguro HTTPS empleando TLS 1.3, evitando ataques de interceptación \textit{(Man-in-the-Middle)}.
		\end{itemize}
	\end{itemize}
	
	\item \textbf{Exposición de Datos Sensibles y de Pago}
	\begin{itemize}
		\item Vulnerabilidad: Interceptación o filtrado de información financiera de los usuarios, direcciones exactas de origen/destino o datos personales de registro.
		\item Medida de Mitigación:
		\begin{itemize}
			\item Cifrado en Reposo y Tránsito: Uso de algoritmos de cifrado simétrico avanzado AES-256 para los datos almacenados en la base de datos (como el historial de servicios y direcciones habituales).
			\item Tokenización de Pagos: La plataforma no almacenará números de tarjetas de crédito o débito en sus servidores locales. Se utilizará una pasarela de pago externa certificada que tokeniza las transacciones, cumpliendo de forma estricta con el estándar internacional PCI-DSS.
		\end{itemize}
	\end{itemize}
	
	\item \textbf{Manipulación y Falsificación de Evidencia Fotográfica}
	\begin{itemize}
		\item Vulnerabilidad: Carga de imágenes falsas, preexistentes o descargadas de internet en lugar de capturas reales del estado de la carga en los puntos de origen y descarga, invalidando el sistema de respaldo ante reclamos.
		\item Medida de Mitigación:
		\begin{itemize}
			\item Control de Captura por Hardware: Restricción en la aplicación móvil para que las fotografías del estado de los objetos solo puedan tomarse directamente abriendo la cámara nativa del dispositivo a través de la app. Se bloqueará por completo la opción de seleccionar o subir imágenes provenientes de la galería local del teléfono.
			\item Metadatos y Marcas de Agua: Almacenamiento automático en el servidor de los metadatos EXIF de cada imagen (fecha, hora y coordenadas GPS de la captura) emparejándolos con el estado del traslado para verificar la consistencia temporal y geográfica.
		\end{itemize}
	\end{itemize}
		
\end{enumerate}

\section{Pruebas de calidad}

Para asegurar que los requerimientos funcionales y de calidad se cumplan sin introducir errores en el ecosistema digital, se establece un enfoque de pruebas en múltiples niveles: 

\begin{enumerate}
	\item \textbf{Pruebas unitarias (unit tests)} 
	\begin{itemize}
		\item \textbf{Objetivo:} Verificar de forma aislada el comportamiento de los componentes mínimos de código, funciones lógicas y clases específicas.
		\item \textbf{Componentes a evaluar:}
		\begin{itemize}
			\item Módulo de cálculo de tarifas flexibles basado en la distancia 
			\item Función de validación de estructura de tokens JWT 
			\item Lógica de cálculo del porcentaje de comisión de la plataforma.
		\end{itemize}
		\item\textbf{Descripción General de la Prueba:} Mediante el uso de herramientas de testeo automatizado en el entorno de desarrollo, se inyectan valores de entrada simulados (tanto válidos como erróneos o límites) directamente a las funciones del backend y frontend para comprobar de manera aislada que reaccionen exactamente según la lógica programada sin afectar otros módulos. 
		\item \textbf{Criterio de Aceptación:} El 100\% de las pruebas unitarias clave deben ejecutarse de forma exitosa. Se requiere una cobertura de código mínima del 80\% mediante herramientas de análisis de cobertura automatizadas.
	\end{itemize}
	\item\textbf{Pruebas de integración (integration tests)}
	\begin{itemize}
		\item \textbf{Objetivo:} Garantizar la correcta comunicación, intercambio de datos y sincronización entre los diferentes módulos del sistema y servicios de terceros.
		\item \textbf{Componentes a Evaluar:}
		\begin{itemize}
			\item Interacción entre la API de la aplicación móvil y la base de datos centralizada al guardar el historial de servicios
			\item Persistencia de imágenes cargadas por Multer en el servidor de archivos asociado al ID del traslado 
			\item Flujo de comunicación síncrona en el chat interno mediante WebSockets.
		\end{itemize}
		\item \textbf{Descripción general de la prueba:} Se ejecutan llamadas de prueba entre subsistemas combinados para verificar que los datos se transmitan, transformen y almacenen correctamente. Se valida que la información viaje cifrada por canales seguros, que las conexiones concurrentes de mensajería mantengan la sincronización mutua y que la persistencia de archivos de evidencias guarde simultáneamente los registros correspondientes en la base de datos sin romper la integridad referencial. 
		\item \textbf{Criterio de Aceptación:} Los módulos deben integrarse sin lanzar excepciones de comunicación o inconsistencias en la base de datos. Los tiempos de respuesta de la API en condiciones normales deben situarse por debajo de los 500ms.
	\end{itemize}
	\item \textbf{Pruebas de sistema (system tests)}
	\begin{itemize}
		\item \textbf{Objetivo:} Evaluar el comportamiento global de la plataforma ejecutando flujos de trabajo completos simulando escenarios reales de uso cotidiano de punta a punta.
		\item\textbf{Componentes a Evaluar: Ciclo de vida completo de un flete : }
		\begin{enumerate}
			\item Cliente solicita flete 
			\item Transportista recibe alerta e inicia viaje
			\item Registro fotográfico de origen 
			\item Seguimiento GPS en ruta 
			\item Registro fotográfico de destino 
			\item Pago electrónico y descuento de comisión 
			\item Calificación final.
		\end{enumerate}
		\item \textbf{Descripción General de la Prueba:} Un evaluador de calidad ejecuta el caso de uso principal de la plataforma simulando el comportamiento de los distintos actores de manera secuencial en un entorno de pruebas controlado. Se verifica visual y lógicamente que el flujo de pantallas sea continuo y que cada acción del usuario desencadene el cambio de estado correcto en el sistema global.
		\item  \textbf{Criterio de Aceptación:} El flujo completo debe ejecutarse sin caídas del sistema ni pérdida de persistencia de datos. No deben existir errores críticos de interfaz que impidan completar la transacción.	
	\end{itemize}
	\item\textbf{Pruebas de Rendimiento, Seguridad y Usabilidad}
	\begin{itemize}
		\item\textbf{Objetivo:} Evaluar las capacidades y restricciones globales del software bajo escenarios críticos de estrés, verificar la resistencia ante accesos no autorizados y medir la ergonomía y usabilidad de las interfaces móviles y web.
		\item \textbf{Componentes a Evaluar:} 
		\begin{itemize}
			\item Motor de mapas en tiempo real y su tasa de refresco
			\item Middleware perimetral de control de accesos
			\item Formularios de autenticación contra inyecciones lógicas 
			\item Arquitectura de navegación de las interfaces de usuario.
		\end{itemize}
		\item \textbf{Descripción general de la prueba:} Descripción General de la Prueba: Se ejecutan de manera simultánea simulaciones automatizadas de carga masiva de peticiones telemétricas para estresar el backend, auditorías de seguridad perimetral intentando forzar peticiones modificando cabeceras de red o omitiendo credenciales, y sesiones de interacción pasiva con usuarios reales ajenos al desarrollo para evaluar los flujos de pantallas.
		\item \textbf{Criterio de Aceptación:} El sistema debe procesar la geolocalización en un máximo de 2,67 segundos bajo la carga de 1.000 usuarios concurrentes, permitir la contratación del servicio en no más de 4 pasos intuitivos, y rechazar de forma estricta el 100\% de las solicitudes no autenticadas mediante códigos de error HTTP 401 o 403.
	\end{itemize}
\end{enumerate}

\section{Prácticas de codificación segura y estándares de calidad }

\subsection{Prácticas de codificación segura} 

El desarrollo del software integra el estándar \textbf{OWASP Top 10} para establecer una línea base de seguridad desde la arquitectura. La estrategia de protección se fundamenta en los siguientes pilares de programación defensiva: 

\begin{itemize}
	\item \textbf{Validación y Sanitización de Entradas (Prevención de Inyecciones):} Se establece como política mandatoria que todo dato proveniente de actores externos sea interceptado y normalizado antes de interactuar con las capas lógicas. Esta capa de abstracción neutraliza intentos de inyección y previene la ejecución de scripts no autorizados, garantizando que el backend solo procese datos estructurados y seguros.
	\item \textbf{Control de Acceso Basado en Roles (RBAC):} La gestión de identidades utiliza un modelo centralizado de autorización donde cada solicitud es auditada por un componente perimetral. Esto asegura que, independientemente del nivel de acceso, el sistema verifique los privilegios del emisor antes de liberar funciones administrativas, tarifas o documentos sensibles, garantizando el principio de "menor privilegio".
	\item \textbf{Políticas de Gestión de Archivos y Evidencias:} Dado que la plataforma gestiona documentación legal y registros fotográficos, el subsistema de carga opera bajo restricciones preconfiguradas: validación estricta de formatos (MIME-Type), límites de transferencia para mitigar ataques de denegación de servicio (DoS) y el renombrado aleatorio de archivos para ofuscar la topología interna del servidor, protegiendo así la integridad de la infraestructura.
\end{itemize}

\subsection{Estándares de Calidad del Producto (ISO/IEC 25010) }

Para la evaluación técnica y aseguramiento de la excelencia del producto, se adopta el estándar internacional \textbf{ISO/IEC 25010.} Este marco define los atributos clave necesarios para que la plataforma sea competitiva y robusta:

\begin{enumerate}
	\item \textbf{Adecuación Funcional:} Esta característica asegura que el sistema cumpla con la totalidad de los requerimientos especificados. Se garantiza la precisión del motor de emparejamiento geográfico y la exactitud en el procesamiento de transacciones financieras, cumpliendo fielmente con las reglas de negocio, lo que genera confianza tanto en Clientes como en Transportistas.
	\item \textbf{Eficiencia de Desempeño:} La arquitectura está dimensionada para responder con fluidez ante la concurrencia masiva. Se ha definido un umbral de \textbf{2,67 segundos} para el procesamiento de datos telemétricos y la actualización cartográfica, asegurando que el seguimiento en tiempo real sea inmediato y sin degradar la capacidad de cómputo del servidor.
	\item \textbf{Usabilidad (Operabilidad):} El diseño de la interfaz prioriza la experiencia del usuario (UX) mediante principios de consistencia y simplicidad. Se garantiza que un usuario pueda completar la contratación de un flete en un máximo de cuatro pasos, implementando una gestión de excepciones que traduce errores técnicos en mensajes informativos, reduciendo así la carga cognitiva del usuario.
	\item \textbf{Fiabilidad (Tolerancia a Fallos):} Ante la inestabilidad de las redes móviles en ruta, el software implementa mecanismos de resiliencia. El sistema preserva los datos de trazabilidad localmente durante la desconexión y sincroniza automáticamente el historial completo al restaurarse el enlace, protegiendo la integridad referencial y evitando la pérdida de información del viaje.
	\item \textbf{Mantenibilidad:} Se impone una arquitectura de \textbf{estricta modularidad} para asegurar la longevidad del sistema. Al separar la lógica de operación de las capas de presentación visual, se facilita la corrección de errores y futuras actualizaciones sin introducir regresiones. Esta estructura no solo reduce la deuda técnica, sino que permite que la plataforma sea altamente escalable ante nuevas demandas del mercado.
	
\end{enumerate}
